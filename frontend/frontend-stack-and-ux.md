# 前端架构与 UX 技术栈总结

这是一套适合中后台、AI 工作台、任务流平台的前端架构方案。本文只提炼架构信息，不包含任何项目业务实现细节。

## 1. 核心技术栈

### 基础框架

- Vue 3
- TypeScript
- Vite
- Vue Router
- Pinia

### UI 与样式

- Arco Design Vue
- Less
- 基于 Design Token 的 CSS 变量体系

### 工程与请求层

- Axios
- ESLint
- Prettier
- `vue-tsc`

### 可选增强

- `marked`
  适合承载 Markdown / AI 生成文本渲染。
- `@vueuse/core`
  适合补充组合式逻辑能力。
- `vue-virtual-scroller`
  适合超长列表、素材库、日志流等高性能滚动场景。

## 2. 推荐架构分层

推荐采用以下目录分层：

```text
src/
├─ api/         # 接口封装层
├─ assets/      # 静态资源
├─ components/  # 通用组件
├─ router/      # 路由层
├─ store/       # 全局状态
├─ styles/      # 全局样式与页面样式
├─ types/       # 类型定义
├─ utils/       # 通用工具
└─ views/       # 页面层 / 业务容器
```

分层职责建议如下：

- `views/`
  承担页面编排、业务流程、数据拼装。
- `components/`
  承担可复用 UI 与交互组件，不直接耦合具体业务。
- `api/`
  统一封装接口，不夹杂页面状态逻辑。
- `store/`
  只放跨页面共享状态，不承载页面私有状态。
- `types/`
  统一管理接口类型、领域模型、组件入参出参类型。
- `utils/`
  放日期、下载、格式转换、请求辅助等纯工具能力。

## 3. 路由架构

推荐采用：

- `Vue Router 4`
- 顶层布局路由 + 子路由业务页面
- 基于路由守卫的登录态校验
- 路由级懒加载

适合的典型结构：

- 登录页
- 主工作台布局
- 多业务模块子页面
- URL 驱动的筛选、详情、面板切换

## 4. 状态管理架构

推荐采用 `Pinia`，并坚持轻量 store 原则。

适合进入全局状态的内容：

- 用户信息
- 登录态 / Token
- 全局偏好设置
- 跨页面共享筛选条件

不建议进入全局状态的内容：

- 页面内部弹窗开关
- 单页表单草稿
- 仅局部组件需要的数据

建议模式：

- 页面状态优先本地化
- store 只做共享状态与少量共享行为
- 请求结果尽量在页面层或组合函数层消费，不把所有接口数据都堆进 store

## 5. 请求层架构

推荐基于 `Axios` 建统一请求实例。

建议统一收敛以下能力：

- `baseURL`
- 超时控制
- Token 注入
- 请求/响应拦截器
- 通用错误处理
- 统一返回结构拆包

推荐目标：

- 页面不直接写裸请求
- 鉴权逻辑不散落在各页面
- 后端返回结构变化时，改动集中在请求层

## 6. UI 架构

推荐以 `Arco Design Vue` 作为主 UI 体系。

它适合的产品形态包括：

- 中后台系统
- 任务流平台
- AI 工具工作台
- 数据管理与资产管理平台

推荐重点使用的组件类别：

- 表单类：表单、输入框、下拉框、选择器
- 反馈类：消息、弹窗、确认框、加载态
- 数据展示类：表格、列表、描述、分页
- 状态表达类：标签、进度、步骤条
- 上传与媒体类：上传、图片、预览容器
- 辅助类：提示气泡、头像、空间布局

## 7. UX 基础设施建议

除了组件库本身，建议在架构层预置几类通用 UX 基础设施：

- 全局加载组件
- 空状态组件
- 媒体预览弹窗
- 文件/字幕/内容预览弹窗
- 全局消息与确认交互规范
- 上传进度反馈容器
- 状态标签映射体系

这些能力一旦沉淀成基础组件，后续页面会非常稳定，且能保持交互一致性。

## 8. 页面结构模式

这套架构更适合“工作台型”页面，而不是官网型页面。

推荐的页面模式：

- 顶部导航 + 主内容区
- 左侧列表 + 中间编辑区 + 右侧结果区
- 侧边栏 + 主面板
- Tabs 驱动的子模块切换
- 配置区 + 任务区 + 结果区

这类结构特别适合：

- 多步骤任务流
- 配置驱动型系统
- 内容生产工具
- 资产管理平台

## 9. 样式架构

推荐采用：

- `Less` 作为样式预处理器
- 全局样式入口
- 页面级样式文件独立维护
- 基于组件库 Token 的 CSS 变量主题体系

建议约定：

- 布局、间距、局部视觉增强写在页面级样式里
- 颜色、边框、文本层级尽量复用 Design Token
- 少写全局污染样式
- 通用样式抽到基础层，避免页面间复制

## 10. 类型系统建议

推荐 TypeScript 全量参与以下层级：

- API 返回类型
- 页面状态类型
- 组件 Props / Emits
- 领域模型类型
- 工具函数入参和返回值

建议目标：

- 页面不依赖“猜字段”
- 接口变化能尽快暴露类型错误
- 组件边界清晰

## 11. 代码规范与质量保障

推荐最小工程规范：

- ESLint 负责行为约束
- Prettier 负责格式统一
- `vue-tsc` 负责类型检查

建议至少约束：

- 禁止 `var`
- 强制 `===`
- 减少未使用变量
- 保持统一引号、分号、换行风格

## 12. 适合复用到新项目的最小组合

如果另一个项目也想沿用这套方案，最小可复用组合可以是：

- Vue 3
- TypeScript
- Vite
- Vue Router
- Pinia
- Arco Design Vue
- Less
- Axios
- ESLint
- Prettier

如果新项目偏 AI 应用或内容平台，再按需补：

- `marked`
- `@vueuse/core`
- `vue-virtual-scroller`

## 13. 一句话结论

这套前端方案本质上是一套面向中后台与 AI 工作台的 Vue 架构，核心优势不是“炫技”，而是分层清晰、组件体系稳定、交互反馈完整、适合快速搭建复杂业务页面。
